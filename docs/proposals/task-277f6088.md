# Proposal: Fix Broken Test Suite Detection

**Task**: task-277f6088

**Project**: t3code-ludics
**Priority**: A
**Effort**: small

## Goal

Ensure the ludics health check system correctly identifies and runs the t3code-ludics test suite so that all 700+ tests are reported as passing instead of 101 tests failing with import errors.

## Acceptance Criteria

- `config.yaml` in the harness has `test_command: "bun run test"` for the `t3code-ludics` project entry
- Running `ludics health-check --project t3code-ludics --force` reports all tests passing
- `mag/test-health.json` records `passed: true` for `t3code-ludics`

## Context

The t3code-ludics monorepo uses **Bun as the package manager** and **vitest as the test runner**, coordinated through **Turbo**. The test pipeline is:

```
bun run test  →  turbo run test  →  vitest (per package)
```

The six packages under test are: `contracts` (70 tests), `shared` (41), `desktop` (31), `scripts` (20), `web` (538), `server` (596).

All test files use `@effect/vitest` for Effect-typed test assertions. This package's exports are vitest-specific — they rely on vitest's internal module resolution and are incompatible with Bun's native `bun test` runner.

The ludics health check's `detectTestCommand` function (in `~/ludics/src/health.ts`, around line 28) auto-detects `bun test` when it finds a `bun.lock` file. This heuristic is correct for many Bun projects, but misleading here: `bun.lock` indicates Bun is the **package manager**, not the **test runner**. The actual test runner is vitest, invoked via `turbo run test`.

Running `bun test` directly produces:

```
SyntaxError: Export named 'assert' not found in module
  '.../node_modules/.bun/@effect+vitest@.../index.js'
101 tests failed (159 pass, 101 fail, 101 errors)
```

Running `bun run test` (the correct command) delegates to `turbo run test` → vitest and all 700+ tests pass.

## Approach

This is a **one-line fix in the harness `config.yaml`**, not in the t3code-ludics project itself. The t3code-ludics project has the correct test setup; the issue is that the harness health checker selects the wrong command.

**File to change**: `/Users/lukstafi/self-improve/harness/config.yaml`

**Change**:
```yaml
# Before:
  - name: t3code-ludics
    repo: lukstafi/t3code-ludics
    path: ~/t3code-ludics
    issues: true
    priority: true

# After:
  - name: t3code-ludics
    repo: lukstafi/t3code-ludics
    path: ~/t3code-ludics
    issues: true
    priority: true
    test_command: "bun run test"
```

The `test_command` field overrides the auto-detected command, routing health checks through the correct `bun run test` → turbo → vitest pipeline.

**Secondary improvement** (out of scope for this task, tracked separately): The `detectTestCommand` heuristic in `~/ludics/src/health.ts` could be improved to check for `turbo.json` or vitest config presence alongside `bun.lock`, to avoid this class of false detection for turbo monorepos.
